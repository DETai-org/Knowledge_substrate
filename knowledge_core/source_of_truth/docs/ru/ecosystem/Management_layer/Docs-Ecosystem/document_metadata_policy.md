## Тип документа `ecosystem`

Тип документа `ecosystem` определяет **специфическую структуру метаданных**, которая включает свои обязательные поля и правила валидации. Важно понимать, что **структура метаданных зависит от типа документа**. В рамках одной **схемы** могут существовать различные типы документов, каждый из которых имеет свою уникальную структуру метаданных. При этом схемы могут сосуществовать в одной SQL‑базе данных, но каждая схема будет оптимизирована под свою предметную область, и её таблицы, поля и связи будут соответствовать специфике конкретного типа документа.

Такой подход обеспечивает:

- **явную границу ответственности каждой схемы** в базе данных,
    
- **совместимое сосуществование разных типов документов** в одной SQL‑структуре,
    
- и позволяет каждой схеме быть оптимизированной под свою область знаний.

**Важно:** Если будет использован **другой тип документа** (например, `project` и другие), для таких документов потребуется **отдельная структура метаданных** и соответствующая политика.

Полная структура:
```yaml

  scope:

  id:

  document_links:  
    - { schema: "", link_type: "", linked_document_id: "" }  
    - { schema: "", link_type: "", linked_document_id: "" }  
---
```

## Классификация

**`Classification`** — группа полей, которые определяют место документа в архитектуре экосистемы, его уровень действия и смысловую нагрузку. Эти поля обеспечивают контекст для поиска и обработки информации. Эта группа полей содержит следующие поля:

### Поле `scope`

Поле **`scope`** определяет **горизонталь действия**: уровень системы или конкретный кластер экосистемы, на который распространяется действие документа.

### Поле `context`

Поле **`context`** определяет **вертикаль действия**: конкретную рабочую среду (*environment*) или функциональную область внутри заданного уровня системы (`scope`). Если `scope` отвечает на вопрос «ГДЕ?», то `context` уточняет «В КАКОЙ ИМЕННО СРЕДЕ?».

Набор допустимых значений `context` напрямую зависит от выбранного `scope`. Контекст должен быть релевантен уровню системы: например, `context: clickup` невозможен при `scope: DET_cluster`, так как инструменты существуют на своём собственном уровне.

**Допустимые значения в зависимости от `scope`:**

---

#### Для `scope: ecosystem`

`management-layer` — общесистемный уровень управления, отвечающий за координацию кластеров, формирование стратегии развития и обеспечение её практической реализации. Этот слой работает с тем, что определено в Governance, и переводит стратегию в структуру и действия.

---

#### Для `scope: DET_cluster`

`management-layer` — локальный уровень управления кластером DET. Обеспечивает координацию развития теории, методологии и практики диалектической экзистенциальной терапии в рамках общей стратегии экосистемы.  
`⚠️По факту пока ещё не существует Но по мере роста будет⚠️`

---

#### Для `scope: DETai_cluster`

`management-layer` — локальный уровень управления кластером DETai. Определяет стратегию развития технологического направления, инициирует и сопровождает запуск проектов, обеспечивает финансовую, юридическую и организационную устойчивость кластера.

`platform` — инфраструктурный и продуктовый уровень кластера DETai. Platform DETai выступает как зонтичный бренд, объединяющий все проекты и цифровые продукты под единой архитектурой. Она не управляет проектами как административная структура, а обеспечивает их интеграцию и единую идентичность.

`uli` — внутренняя интеллектуальная среда разработки (Universal Learning Intelligence). U.L.I. представляет собой концептуальную модель среды, в которой люди и AI совместно учатся, структурируют знания и создают продукты. Это операционная среда команды, где человек формулирует смысл и направление, а AI помогает превращать их в структуру, документацию и работающие решения.

#### Для `scope: Tools`

На уровне инструментов контекстом является конкретный инструмент экосистемы. Каждый инструмент выполняет свою функцию в общей системе и может быть описан в соответствующей документации.

`github` — система контроля версий и репозиториев кода. Используется для хранения кода, управления пулл-реквестами и автоматизации процессов через GitHub Bot и GitHub Projects.

`linear` — система управления задачами и проектами для разработки. Обеспечивает отслеживание задач, статусов, приоритетов и интеграцию с GitHub через связь с пулл-реквестами.

`clickup` — универсальная система управления задачами и проектами для всех команд экосистемы. Позволяет координировать работу и отслеживать прогресс.

`slack` — платформа для коммуникации команд. Обеспечивает уведомления о событиях в GitHub и Linear, синхронизацию и оперативное взаимодействие.

*(Список инструментов может расширяться по мере развития экосистемы. Каждый новый инструмент получает свой контекст и описание.)*

---

#### Для `scope: Infrastructure`

На уровне инфраструктуры контекстом являются конкретные инфраструктурные компоненты, обеспечивающие работу экосистемы.

`servers` — серверная инфраструктура, обеспечивающая функционирование всех систем.

`Knowledge_Substrate` — База данных SQL и библиотека знаний МК Докс


---

#### Для `scope: STORYTELLING`

На уровне сторителлинга контекстом являются форматы контента и каналы коммуникации с внешней аудиторией.

`website` — контент для сайта экосистемы, внешние коммуникации.

`social` — контент для социальных сетей.

`articles` — статьи и публикации.

`presentations` — презентационные материалы.

#### Принцип соответствия `scope` и `context`

При выборе значений для метаданных документа необходимо соблюдать следующую логику:

1. Сначала выбирается **`scope`** — уровень системы, к которому относится документ.
2. Затем выбирается **`context`** — конкретная среда или область внутри этого уровня.
3. Контекст должен быть **релевантен** выбранному уровню (например, контексты инструментов используются только при `scope: Tools`).
4. По мере развития экосистемы список допустимых контекстов может расширяться. Новые контексты должны быть описаны и зафиксированы тут.

### Поле `layer`

Поле **`layer`** определяет **смысловой слой** документа в архитектуре системы, отвечая на вопрос "какого рода знание здесь содержится?".

По сути, `layer` выполняет функцию **кластеризации** для поля `function`: каждый слой объединяет несколько типов документов со схожей смысловой нагрузкой.

**Допустимые значения:**

`philosophy` — слой принципов, фундаментальных концепций и ценностей системы. Сюда относятся документы, которые задают мировоззренческую рамку и отвечают на вопрос "зачем?".
*В этом слое используются например такие типы документов (`function`): philosophy и principles*

`architecture-and-logic` — слой описания архитектуры, логики работы и взаимосвязей компонентов. Документы этого слоя объясняют, как устроена система и почему она устроена именно так.

`technical-standards` — слой технических стандартов, норм, спецификаций и правил. Документы этого слоя фиксируют договорённости и регламенты, которым необходимо следовать.


📌 _Важно:_ Документы с функциями `guide`, `how-to` и `tutorial` относятся к **Onboarding** (каналу входа и обучения) и не входят в основные слои `layer`. Для таких документов в поле `layer` указывается значение `null`. Подробнее об архитектуре слоёв см. в  [Документационная архитектура экосистемы](https://detai-org.github.io/Knowledge_substrate/ru/ecosystem/Management_layer/Docs-Ecosystem/documentation-architecture/).

### Поле `function`

Поле **`function`** описывает **функциональную роль документа**, то, как он должен использоваться. На данный момент определено **12 основных функций**:

Основные функции:

> Полное описание этих функций приведено в документе: [Функция документов](https://detai-org.github.io/Knowledge_substrate/ru/ecosystem/Management_layer/Docs-Ecosystem/functions_of_documents/)

#### Специальное правило для `function: index`

Функция `index` используется **только** для файлов `index.md`, которые являются входными страницами разделов.

Обязательные условия для `index`-документов:

- `layer: null`
- `id` формируется как комбинация базового идентификатора раздела и суффикса `-index` (например: `governance-index`), чтобы избежать повторяющихся ID и разных интерпретаций при именовании index-файлов.

## Описание

**`Descriptive`** — раздел, содержащий базовые идентифицирующие и описательные метаданные документа.

### Поле `id`

Поле **`id`** — уникальный идентификатор документа. Представляет собой название файла в стиле `kebab-case`. По сути, имя файла документа и является его уникальным ID.

Для файлов `index.md` рекомендуется использовать правило уникализации:

```yaml
id: <section-base-id>-index
```

Это позволяет избежать коллизий между разными `index.md` в разных директориях.


### Поле `version`

Поле **`version`** фиксирует текущую редакцию документа. Версия изменяется при внесении значимых изменений в содержание.

Пример:
```yaml
version: v1
version: v2
```

Для документов с `function: philosophy` используется формат изданий (edition), в соответствии с процессом версионности:

```yaml
version: 1st-edition
version: 2nd-edition
version: 3rd-edition
```

То есть для философских документов применяется не `v1/v2`, а формат «N-th edition».

### Поле `status`

Поле **`status`** определяет жизненный цикл документа и его актуальность для использования.

**Допустимые значения:**

`draft` — документ находится в разработке, требует доработок и проверки, не подлежит применению.

`active` — документ завершен, актуален и является официальным источником для использования в рабочих процессах.

`deprecated` — документ устарел и больше не поддерживается; рекомендуется использовать более новые версии или заменяющие документы.

`archived` — документ больше не используется и перемещен в архив. Сохранен для истории, но не актуален для текущих процессов.

### Поле `date_ymd`

Поле **`date_ymd`** — дата последнего значительного изменения или создания документа в формате ГГГГ-ММ-ДД.

## Ссылки

**`Links`** — раздел для указания всех связанных ресурсов, как внешних, так и внутренних.

📌 _Примечание :_ В большинстве случаев эти поля требует отдельной аналитической работы, поэтому при создании документа здесь практически всегда проставляется значение `null`. Ссылки добавляются постфактум, по мере выявления и фиксации смысловых связей между документами.

### Поле `external_links`

Поле **`external_links`** содержит массив внешних ссылок. Каждая ссылка должна быть описана с указанием типа ресурса.

Пример:
```yaml
external_links:
  - { type: "website", url: "https://example.com" }
  - { type: "telegram", url: "https://t.me/example" }
  - { type: "figma", url: "https://figma.com/file/..." }
```


### Поле `document_links`

Поле **`document_links`** содержит массив ссылок на другие документы в единой SQL-базе знаний экосистемы. Каждый документ в базе относится к определённой схеме (например, `publications_schema`, `ecosystem_schema`, `project_schema`), и для каждой схемы определены свои типы документов.

Ссылка на документ включает три параметра:

- **`schema`** — тип документа, определяющий, в какую схему заглянуть (например, `publications_schema`)
- **`link_type`** — тип конкретного документа в схеме (например, `post`, `research_publication`, `quote`)
- **`linked_document_id`** — уникальный идентификатор документа в базе (первичный ключ)

Пример:
```yaml
document_links:
  - { schema: "publications_schema", link_type: "post", linked_document_id: "unique_id_123" }
  - { schema: "publications_schema", link_type: "research_publication", linked_document_id: "unique_id_456" }
  - { schema: "project_schema", link_type: "project", linked_document_id: "unique_id_789" }
```

📌 *Примечание 1:* Подробнее о схемах документов см. в соответствующей директории: https://github.com/DETai-org/Knowledge_substrate/tree/main/knowledge_core/source_of_truth/schemas/publications

## Принцип иерархии документов

Документы с более широким `scope` (например, `ecosystem`) задают стандарты и правила для документов с более узким `scope` (например, `DETai_cluster`). Аналогично, документы с более высоким `layer` (например, `philosophy`) определяют рамки для документов нижележащих слоев (`architecture-and-logic`, `technical-standards`).


## Принцип развития системы

Метамодель, описанная в данном документе, может расширяться по мере развития экосистемы. Добавление новых полей, значений в существующие поля или изменение классификации должно сопровождаться обсуждением и фиксироваться **новой версией** данного документа.
