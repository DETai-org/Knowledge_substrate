# Как быстро вникнуть в проект

Как быстро вникнуть в проект напарника (через Агент + проверка на сервере)

## 🎯 Цель

За 1–2 итерации получить рабочий контекст по незнакомому проекту и Начать делать в нем точечные исправления не ломая всё то что уже работает

---

## 1) Запускаем режим **Агента** для первичного анализа репозитория

Сформулируй запрос к Агенту (подключённому к GitHub) так, чтобы он **одним проходом** собрал контекст:

**Пример запроса к Агенту**

> Проанализируй `sandbox/tools/telegram_bots/birthdays_bot` и всё, что с ним связано. Объясни, как бот связан с `sandbox/VK/birthdays` (какие функции вызываются), где находится точка входа и как его запускать. Покажи, где хранятся настройки (`config_bot.toml`), какие модули критичны для запуска (handlers, settings, services), и что потребуется, чтобы перевести запуск на systemd (модульный путь для ExecStart и рабочая директория).

**Что получить на выходе**

- Структурированный отчёт: ключевые файлы, точка входа (модуль для `-m`), связь бота с подпроектом (какие функции вызывает), где лежит конфиг.
    
- Короткие «must-have» проверки (существуют ли пути из конфига и т.д.).

![Pasted image 20250927213705.png](Pasted image 20250927213705.png)

Сохраняем отчёт (например, `Анализ ботов и скриптов.*`) и **целиком вставляем его сюда, в GPT-чат**.

> [!WARNING]
Количество таких запросов в режиме агента ограничено на месяц поэтому экономно с этим режимом 


## 2) Делаем «снимок» директории на сервере

На сервере сохрани «живую» структуру проекта, путь ~/projects или любой другой если точно знаешь что элементы не разбросаны по разным папкам:

```bash
cd ~/projects
tree -L 3 > Directory_structure.txt
```

---

## 3) Передаём материалы ассистенту (GPT)

Сразу после «снимка» директории **накидываем весь контекст в чат**:

1. Вставь **полный отчёт Агента** по репозиторию (тот, что он сгенерил на основе анализа репрозитория GitHub).
    
2. **Прикрепи `Directory_structure.txt`** с сервера (живое дерево каталогов).
    
3. Добавь короткий запрос:
    

```text
Вот отчёт и живая структура директорий.
Сформируй итог по точке входа, связям подпроектов и пути запуска через,
а затем назови минимальный список файлов, которые мне тебе прислать для финальной проверки.
```

> Зачем: отчёт Агента даёт «логическую» картину, `Directory_structure.txt` — «физическую». GPT сверяет их и понимаю, где правда, после чего просит у тебя ровно те файлы, которые нужны.

---
 Запрашиваем у GPT **минимальный набор файлов** для финальной проверки

После этого **присылай ровно эти файлы** — я проверю импорты, пути, точку входа 


## 5) Формируем итоговые задачи

Когда у GPT уже есть **полное понимание структуры проекта, точек входа и связей**, можно формировать **любые задачи**.  
Например:

- создание systemd-юнита для запуска бота,
- настройка синхронизации,
- правки конфигов,
- или интеграция подпроектов.
    

Важно: после этапа анализа работа идёт **по UZI** (узкому запросу на исполнение) — формируем чёткую, локальную задачу и уже её передаём в Codex.

---

> [!NOTE]  
> 🔑 Самое важное:
> 
> 1. **Сначала** нужно установить взаимосвязи (где точка входа, какой подпроект, какие конфиги).
>     
> 2. **Потом** — любая задача будет корректной, потому что уже есть контекст.
