# ADR-2026-02-11 — Использование Airtable как временного working-context слоя для ChatGPT

## Статус
Принято

## Дата
2026-02-11

## Контекст

В экосистеме DETai уже существует централизованная система знаний:
- Knowledge Core (GitHub / Markdown)
- Knowledge Storage (SQL)
- Формализованные метаданные (version, status, taxonomy, policies)
- Единый Source of Truth (SoT)

В настоящий момент агентский слой (Agent / Tool), обеспечивающий доступ ChatGPT к Knowledge Storage по запросу, **ещё не реализован**.

Параллельно существует практическая необходимость:
- регулярно поднимать контекст в чатах ChatGPT,
- использовать стандарты, политики и рабочие документы,
- избегать ручной загрузки файлов в каждый чат,
- не нарушать архитектурный принцип единого Source of Truth.

## Решение

Принято решение **временно использовать Airtable** как инструмент из кластера Knowledge Storage **исключительно в роли working-context слоя**, подключаемого к ChatGPT через официальный коннектор.

Airtable:
- **НЕ является Source of Truth**
- **НЕ содержит полную базу знаний**
- **НЕ синхронизируется автоматически с SQL или GitHub**
- **НЕ заменяет будущий агентский слой**

Использование Airtable ограничено:
- небольшим набором файлов и документов,
- актуальных для текущего этапа разработки,
- применяемых как временный внешний контекст для диалогов с ChatGPT.

## Обоснование

Данное решение принято по следующим причинам:

1. Airtable позволяет быстро подключить структурированные пользовательские данные к ChatGPT без разработки пайплайнов и агентов.
2. Использование Airtable снижает когнитивную и операционную нагрузку на этапе, когда агентская инфраструктура ещё не готова.
3. Airtable используется как **working memory**, а не как база знаний.
4. Решение не создаёт второго Source of Truth и не нарушает архитектурную целостность системы.
5. Airtable органично вписывается в текущий кластер инструментов Knowledge Storage как временный инструмент.

## Альтернативы, которые были рассмотрены

- Немедленная реализация Agent + SQL retrieval  
  → отклонено как слишком большое сейчас не вывезем

- Полная миграция Knowledge Core в Airtable  
  → отклонено как архитектурно некорректное (дублирование SoT).
## Дальнейшие шаги

- Реализация Agent / Tool слоя для доступа ChatGPT к Knowledge Storage (SQL + GitHub).
- Размышление о том может ли он использоваться и тогда как его поддерживать или полностью от него потом отказаться как источника контекста.
- Фиксация миграции в отдельном ADR.
